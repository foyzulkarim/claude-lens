claude-lens — guide & metric reference

This dashboard reads files written by the Claude Code CLI on your machine and shows what you've been doing — sessions, prompts, tool usage, and estimated cost. Everything is local: nothing on this page is sent to a server.

1. Where the data comes from

claude-lens auto-detects your Claude Code data directory (CLAUDE_DIR) and reads four kinds of files. The active path is shown in the header.

File / folderWhat it containsUsed for
history.jsonlEvery prompt you've submitted with timestamp, project, and session IDPrompt History, Project Activity
projects/<encoded>/<sessionId>.jsonlPer-message records: token usage, tool calls, model usedTool Calls, Daily Cost, Summary panels
sessions/*.jsonSession metadata filesExposed via API (not currently rendered)
stats-cache.jsonClaude Code's internal cached statsReturned via API for completeness

Auto-detect order: CLAUDE_DIR env var → ~/.claude → platform-specific paths (Windows %APPDATA%\Claude, macOS ~/Library/Application Support/Claude, Linux $XDG_CONFIG_HOME/claude). The first directory containing any of projects/, history.jsonl, sessions/, or stats-cache.json wins.

2. Today vs All-Time panels

Two side-by-side cards at the top. Today is your local date (see Timezone below). All Time is the sum across every day on disk.

Top row

  • Sessions — distinct sessionId values that produced any user message that day. Sessions that span midnight are counted in each day they touched.
  • Messages — number of user messages (your prompts and tool-result turns) on that day.
  • Tool Calls — number of tool_use blocks emitted by the assistant that day.
  • Cache Hit — share of input tokens that came from cache reads (see Cache Performance). Color-coded: green > 80%, amber > 50%, pink otherwise.

Bottom row

  • Est. Cost — sum of input + output + cache-read + cache-write tokens multiplied by the configured rates.
  • Saved — estimated savings vs a no-cache baseline (see Cache Performance).
  • Cache Read — total tokens served from cache that day.
  • Output Tokens — total tokens the model generated that day.

Top Models

Lists the top three models by number of assistant responses (not tokens). Useful for spotting which Claude variant is doing most of the work.

3. Cache Performance

Anthropic's prompt caching lets long prefixes (system prompt, instructions, files re-read every turn) be billed at a discount on subsequent turns: you pay full price the first time (a cache write) and a tenth of input price thereafter (a cache read).

Cache Hit Rate

hitRate = cacheRead / (input + cacheRead + cacheCreate) × 100

The fraction of all input tokens that came from cache reads. Higher is better. A long-running session with stable context typically reaches 90%+.

Cost With Cache vs Cost Without Cache

costWithoutCache = (allInput × $5/M) + (output × $25/M)
saved = costWithoutCache − actualCost

"Without cache" treats every cached input token as if you paid full input price for it. "Cache Savings" is the difference.

Caveat: the no-cache baseline hard-codes Bedrock ap-southeast-2 rates ($5/M input, $25/M output). If you set RATE_INPUT / RATE_OUTPUT for Anthropic API pricing in .env, your actual cost reflects that, but the "Saved" column still compares against $5/M — so the savings number understates reality on the Anthropic API.

5. Tool Calls

Counts every tool_use block the assistant emitted across all projects, then sorts by frequency. Click any tool row to drill in: the most recent 200 calls are shown with the relevant arguments (Bash command, file path, Grep pattern, etc.) and a timestamp.

The detail panel includes an Export button that downloads all calls (not just the first 200) as a plain-text file — useful for grep'ing through your own command history.

6. Project Activity

Aggregated from history.jsonl. Rows are sorted by message count (most active first):

  • Project — short name + full encoded path (Claude Code stores projects as URL-encoded directory names).
  • Messages — total prompts you've submitted in that project, ever.
  • Sessions — distinct sessions in that project.
  • Last Active — most recent prompt timestamp.

7. Prompt History

Every prompt you've ever sent, newest first. Use the project tabs to filter. Each row shows time, project, the prompt text (hover for the full message), and the first 8 characters of the session ID.

8. How cost is calculated

The server multiplies aggregated token counts by per-million-token rates configured in .env:

cost = input × RATE_INPUT + output × RATE_OUTPUT + cacheRead × RATE_CACHE_READ + cacheCreate × RATE_CACHE_CREATE

(All RATE values are USD per 1,000,000 tokens.)

VariableDefault (Bedrock)Anthropic API equivalent
RATE_INPUT$5.00$15.00
RATE_OUTPUT$25.00$75.00
RATE_CACHE_READ$0.50$1.50
RATE_CACHE_CREATE$6.25$18.75

Override any of these in .env and restart the server. Costs are estimates; the authoritative number is on your bill.

9. Token types explained

  • Input tokens — fresh tokens you sent that weren't cached. Most expensive per token.
  • Output tokens — tokens the model generated. Most expensive overall.
  • Cache create (cache write) — input tokens being placed into the cache for later reuse. Charged at ~125% of input price (a one-time premium).
  • Cache read — input tokens served from cache. Charged at ~10% of input price.

Sums of these come from message.usage on each assistant turn in the project JSONL files.

10. Timezone & date bucketing

All daily aggregations bucket by your local date (your system's IANA timezone, shown in the header). A turn at 11:30 PM local on Monday is on Monday's row even if UTC says it's already Tuesday.

Individual timestamps in tables (Last Active, Prompt History) are rendered with toLocaleString, also in your system timezone.

11. Theme & time format

  • Theme (🌙 / ☀️) — light/dark toggle. Defaults to your OS preference, persisted in localStorage.
  • Time format (12h / 24h) — toggles between 24-hour and AM/PM display for all timestamps. Persisted in localStorage; switching re-renders without re-fetching data.

12. FAQ & limitations

Is anything sent to the network?

No. The server is Express on localhost:3456 and reads files from disk. The HTML page only talks to that server. There are no analytics, no telemetry, no outbound calls.

Why does my cost look different from my Anthropic / AWS bill?

The numbers are computed from the rates you set in .env. They don't account for free tiers, prepay credits, region-specific pricing, or invoicing rounding. Treat them as a directional estimate.

Why doesn't the page auto-refresh?

It doesn't yet. Reload the page to pull the latest data. Files on disk update as Claude Code runs.

I see "No Claude data directory found"

Either Claude Code hasn't been run yet, or the data is somewhere unusual. Set CLAUDE_DIR in .env to the right path. The error banner lists every path the auto-detector tried.

Port 3456 is in use

Another instance of claude-lens is already running. Stop it, or change the PORT constant in server.js.
